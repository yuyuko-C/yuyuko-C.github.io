Game Framework - 生命周期

发表于2022-11-18|更新于2023-03-15|UnityGame Framework
|字数总计:2.5k|阅读时长:8分钟|阅读量:

前排提醒: 阅读本文需要对 Unity 的工作机制有基本的了解, 请勿急于了解框架, 扎实基础才是最重要的事情

如果是你, 你会如何设计 Unity 的游戏框架呢 ?


我们都知道 Unity 的逻辑依赖于 MonoBehaviour. 当 Unity 启动后, 当前场景所有激活物体的所有附带脚本都会走那套固定的生命周期循环. 这套循环会在特定的时间执行 MonoBehaviour 的特定函数. 这些函数就是生命周期函数.

游戏中的每个对象都有其生命周期, 因此生命周期函数会是每个游戏对象都离不开的东西, 不继承自 MonoBehaviour 就无法使用生命周期函数, 但是 MonoBehaviour 过于庞大(这也是它如此方便的原因)每个 MonoBehaviour 脚本的执行都会携带大量变量占据内存, 也会执行许多自带方法占据性能.


如何让游戏中的对象都能使用生命周期函数又不用继承自 MonoBehaviour 就成了我们第一个要解决的问题. 这时就需要我们今天的主角: BaseComponent.

BaseComponent

BaseComponent 主要负责管理生命周期, 并且与游戏进程的生命周期保持一致.

将之继承自 MonoBehaviour 让其成为我们生命周期函数的驱动之源.

不过光有 BaseComponent 还不够, 既然是管理生命周期不能是凭空调用, 还需要被调用的对象.

这个对象可以是任何需要生命周期的对象, 不过更好的办法是由 BaseComponent 负责调用管理类的生命周期函数, 再由各个管理类去管理这些生命周期对象.

管理类会统一管理器其成员, 常作为静态类或者单例类而存在. 通过管理类可以方便访问其成员, 也因统一管理的模式方便内存管理, 因此在游戏中使用管理类是一个程序员必须学会的方法. 他能保住你的头发, 降低你的焦虑, 延长你的寿命, 给你高效便捷的编码体验, 是程序员居家旅行必备之良药, 非常值得推荐, 当然对编写者的要求也会相应的提高.

在概述篇中, 游戏中的各种管理类就已经被E大列举出来了, 既然有这么多管理类, 咱就声明一个抽象类作为管理类的基类 GameFrameworkModule

GameFrameworkModule

再思考

  • Update 为何要设立这样两个参数?

    日常游戏开发中, Update 作为每帧都会调用的函数最离不开的就是帧时长 (elapseSeconds), 在 Unity 中即是 Time.deltaTime. 另外一个则对应的是 Time.unscaleDeltaTime. 这两个变量都与引擎高度绑定, 且为简单的 float 类型. 作为希望脱离游戏引擎而存在的 GF 来说将其作为参数可以让其不被引擎制约.

GameFrameworkEntry

因为这里不将管理类作为单例模式使用, 再声明一个静态类 GameFrameworkEntry 作为管理”管理类”的管理类, 用于操控管理的唯一实例.

再思考

  • GetModule 函数为何对泛型的约束是 class 而不是 GameFrameworkModule?

    因为不希望用户自己创建游戏管理类做了访问限制, 所以将 T 作为接口类型承接管理类,于是T只能约束为class.

  • GetModule 函数当组件不存在时会通过接口反射信息创建对应的游戏模块, 请问有办法优化吗?

    通过预先创建的办法让 GetModule 无法进入创建逻辑, 可以避免反射创建, 缺点就是用不到的模块的内存空间被浪费.

  • 组件轮询用字典存储应该怎么实现?

    字典的轮询是根据键的添加顺序执行的, 所以需要按照从高到低的优先级预先创建所有组件. 不能动态创建, 缺点同上.

  • 是否有种办法能让游戏组件避免使用反射, 还能动态创建, 并且按优先级轮询?

    有. 在使用原用的 GameFrameworkLinkedList<GameFrameworkModule> 的前提下

    1. 给 GameFrameworkModule 添加一个抽象方法获取其接口类型 internal abstract Type InterfaceType { get; }

    2. 声明一个字典 Dictionary<Type, Func<GameFrameworkModule>> 存储接口类型与创建游戏组件的委托的对应关系. 要是再节约内存一点, 还可以创建一个就删除一个, 反正不会创建第二次. 创建完了这个记录字典都能置空给 GC 了.

大功告成, 欧耶!

GameFrameworkComponent

但我们还不能高兴的太早, 众所周知, 当脚本离开了 MonoBehaviour 之后, 脚本运行时的状态将不再可见, 这将不利于项目维护, 也不便发现问题.

项目开发总不是一帆风顺的, 如果你还想省点头发, 这就是我们必须做出的妥协, 所以我们对为每个 GameFrameworkModule 都需要提供一个组件挂载用于显示各种必要的信息.

  • 纵使能飘向更远的地方, 也必须停在中央, 因为我们终须返航. (就™突然有感而发, 请主动忽略这句话. 妥妥的中庸之道)

再度改写 BaseComponent

BaseComponent 作为生命周期管理类本质上也是一个管理类, 所以我们可以对 BaseComponent 再度改写, 让其继承 GameFrameworkComponent.

至此, 我们已经解决了生命周期的问题. 但是 BaseComponent 仅仅解决生命周期还是略显单调, 与其他的管理类的庞大格格不入, 所以给其加入更多游戏引擎提供的配置项, 对游戏进行全局设置. 最终便形成了 UnityGameFramework 里我们看到的 BaseComponent.
