事件中心

事件中心有必须的两个部分, 事件探测器和事件发布中心. 他们好比就是记者和新闻电视台的关系. 记者探测事件, 电视台分发事件, 所有观看电视台的观众都会接收到这个事件. 再也不需要观众们互相告知有没有此事件发生了.

观众想要接收到事件就必须先把电视切换到新闻联播, 不然就会错过事件. 这一行为被称为订阅. 让分发中心知道你对这个内容感兴趣, 当事件发生的时候就会通知你.

GF 中的事件中心

GF 中的事件中心由三部分组成.

  • BaseEventArgs : 所有事件的基类, 提供一个事件ID用于辨识事件.
  • Event : 改名为 EventNode 更好, 用来存储 事件发送者 与 事件信息
  • EventPool : 事件池, 起着事件中心的作用, 用于事件发布.

BaseEventArgs

BaseEventArgs 继承自 GameFrameworkEventArgs, 而 GameFrameworkEventArgs 继承自 System.EventArgs .

而 System.EventArgs 没有任何实际作用, 只是在 System 命名空间下的类而已. 结合在 EventPool 中使用 System.EventHandler 委托, 而 System.EventHandler 的事件参数正是之前用到的 System.EventArgs. 因此我们可以大胆猜测是为了承接系统的事件进行的设计.

为什么要承接系统事件? 我不知道, 猜测是关于IO与网络那块有相应的东西.

至于 GameFrameworkEventArgs 仅是为了让其继承 IReference 接口, 为可回收做准备.

直到 BaseEventArgs 才给出一个抽象方法 Id. 这个 Id 获取到的就是事件的唯一标识.

Event

Event 是处于 EventPool 类中的私有类, 其作用是临时存储 事件发送者 和 事件. 同样继承 IReference 接口, 为可回收做准备. 每当事件发生时就会使用 Event 保存两者的信息.

这个类的使用范围都是在轮询函数 Update 中线程安全的使用事件.

EventPool

作为事件中心, 除了之前讲的一样需要事件探测和事件发布外, 还需要一个收回探测. 因为如果函数所属的对象已经销毁了是需要将其回收的, 不能继续触发. 因此 EventPool 定义了 Subscribe / Unsubscribe 和 Fire 方法. 分别是订阅, 取消订阅 和 事件发布.

EventPool 详解

我们可以看到 EventPool 是一个泛型的非静态类. 正常来说作为事件中心本该是唯一的, 这是怎么回事呢? 我们来逐一分析:

  • 能是静态类吗?

    根据需求来.

    看 EventPool 的构造函数参数 EventPoolMode , 这就决定了不能只有一个 EventPool. 这个参数是对事件的处理函数进行限制, 如果事件池在使用中不符合要求则会抛出异常. E大通过这个参数希望事件池自己对此条件进行检查, 这就注定了必须要有多个池子, 不然这个参数将毫无意义, 所以不将 EventPool 作为静态类使用. 如果不需要的此种条件检查可以将之声明为静态类.

  • 能取消泛型吗?

    根据需求来.

    这里的泛型的作用不是很显眼. 单纯是为了限制事件类型.

    这里涉及到一个访问权限的问题. 但在讲解前我们必须知道一些补充知识 :

    1. EventManager 只是 EventPool 在泛型作为 GameEventArgs 的套壳.
    2. GameEventArgs 继承自 BaseEventArgs, Packet 也继承自 BaseEventArgs, 而这两个都是空继承.
    3. 只有 BaseEventArgs 以及其子类才能用在 EventPool , 因为 BaseEventArgs 这一步才有事件 ID. 所以如果取消泛型, EventPool 里存的只能是 BaseEventArgs.

    我们注意到 BaseEventArgs 的访问权限为 public, EventPool 的泛型限制为 BaseEventArgs, 这意味着所有 BaseEventArgs 的子类都可以被传递进来. 而能被外部通过接口访问的 EventManager 能使用的只有 GameEventArgs , 也就是说E大只希望用户使用的事件中心的事件基类只能是 GameEventArgs.

    而同样能被事件池所接受的 Packet 则从事件伪装成了网络消息, 虽然也是在 EventPool 中调用, 但却完全不能在明面上的事件中心 ( EventManager ) 中使用. 只能在 NetWorkChannel 中使用. 关于 Packet 的事情我们留到网络篇再讲.

    由此可知, E大是故意将其切割成了两条线, 因为是空继承意味着这两个类完全可以被 BaseEventArgs 替代, 这么做的必要性目前尚且不知. 笔者本人才学疏浅不敢乱下定论, 但可以肯定的是如果没有分成这两条线, 这个泛型是可以取消的.
