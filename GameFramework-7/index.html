Game Framework - 对象池

发表于2023-04-25|更新于2023-04-28|C#Game Framework
|字数总计:1.4k|阅读时长:5分钟|阅读量:

Unity 为何需要对象池 E 大已经介绍过了. 我就不再赘述. 但我依稀记得群友”一沙鸥”一直都搞不清楚 RefrencePool (引用池)和 ObjectPool (对象池)有什么区别. 当时已经给她解答过了, 这边再分享给读者. RefrencePool 主要处理的是 new 的部分. ObjectPool 则是对 RefrencePool 的再封装, 是更高级的功能.

RefrencePool ObjectPool
掌管的 new 的部分 不介入的 new 的部分
操作的 System.Object 操作的 UnityEngine.Object

ObjectBase

装载外部 object 的类, 拥有对象池所需的几乎一切属性. 此类不会作为返回值, 仅作为自定义对象池对象的父类继承与泛型约束

  • Name 在对象池的索引
  • Target 容纳的外部对象
  • Locked 控制其是否可以被释放的手动标识
  • Priority 决定资源被释放的优先级
  • LastUseTime 决定了其是否被释放以及被释放的顺序
  • CustomCanReleaseFlag 作用和 Locked 重合, 但其属于不可后天修改的规则标识. 优先级与 Locked 并列
  • 还定义了自定义对象行为的 OnSpawn 和 OnUnspawn, 以及对资源释放的 Release 方法

ObjectPoolBase

ObjectManager.ObjectPool 的父类, 作为非泛型对象池的接口的替代. 如果和我一样是强迫症的可以定义非泛型版本的 IObjectPool 让其继承然后返回此接口. 此类不会被其他类使用但会作为非泛型对象池返回出去, 因此无法作为 ObjectPoolManager 的私有类存在.

ObjectManager.Object

是 ObjectManager 的私有类, 装载 ObjectBase, 通过操控 ObjectBase 间接实现对外部 object 的管理.

基本就是 ObjectBase 的套壳, 多了个 SpawnCount 用来引用计数, 实现 IsInUse 属性和此对象是否可以释放的判断. 其余的方法诸如 Create Spawn 都是对象池调用的方法. 改变 LastUseTime 和 SpawnCount 属性. 顺带调用 ObjectBase 中的 OnSpawn 和 OnUnspawn

ObjectBase 这种用法颇有种 ITaskAgent 的美, 我觉得重命名为 ObjectAgent 或 PoolObjectAgent 之类的名字也不会有违和感.

ObjectManager.ObjectPool

  • m_Objects 使用 ObjectBase.Name 作为 Key 索引到所有同名的 ObjectManager.Object.

  • m_ObjectMap 使用外部 object 作为 Key 索引到承载它的 ObjectManager.Object

  • m_DefaultReleaseObjectFilterCallback 资源释放的默认筛选委托.

  • m_AllowMultiSpawn 是否允许多次获取对象. 此设置与 ObjectManager.Object.SpawnCount 挂钩.

  • m_AutoReleaseInterval 自动释放对象的间隔时间

  • m_Capacity 允许容纳的最大对象数

  • m_ExpireTime 过期时间, 与 ObjectManager.Object.LastUseTime 挂钩.

  • m_Priority 用于决定 Inspector 的显示顺序.

ObjectPoolManager

就是一堆关于 ObjectManager.ObjectPool 的函数, 是否有对象池, 得到对象池, 生成各种对象池的函数. 返回 ObjectPoolBase 和 IObjectPool<T>

AssetObject 与 ResourceObject

填一下在资源管理模块的挖的坑. 这两个分别对应资产和 AB 包.

AssetObject

  • m_DependencyAssets 记录所有依赖的资产, 此数据来源是通过打包数据得到依赖资产名->加载依赖资产->添加到此列表实现的.
  • m_Resource 是记录所属 AB 包的(是真的 AB 包不是装载了 AB 包的 Resource )
  • m_ResourceHelper 和 m_ResourceLoader 不过是卸载资产的时候要用的东西罢了
  • base.Target 则是记录的本资产.

ResourceObject

  • m_DependencyResources 记录所有依赖的 AB 包, 此数据是根据依赖资产获取到的依赖 AB 包, 依赖 AB 包在添加函数中已经做了重复判断.
  • m_ResourceHelper 和 m_ResourceLoader 不过是卸载 AB 包的时候要用的东西罢了
  • base.Target 则是记录的本 AB 包.

知识点图

这期学到了一种新型的权限控制方法. 这是我们首次返回给外部的是类而不是接口. 将类声明为公共的, 方法声明为 internal. 但类仍然和接口一样是抽象的. 所以做到了即使返回给你类也和接口一样的限制.

GF 的对象池不介入对象的产生, 只进行存储和释放, 并且采用引用计数的方式进行, 在资源的释放上使用时间和容量并行的策略, 通过是否允许多次获取来防止不能被同时利用的资源同时利用.
