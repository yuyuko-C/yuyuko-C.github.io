Game Framework - 任务异步

发表于2022-12-08|更新于2023-04-16|UnityGame Framework
|字数总计:4.8k|阅读时长:15分钟|阅读量:

为什么要有异步

正如E大开篇所说

为了保证玩家的体验，我们不推荐再使用同步的方式加载资源.

对于比较耗时的任务, 同步执行方法因为需要当即返回结果, 会造成主线程卡顿. 所以, 异步的处理方案同样是游戏开发中绕不开的一个槛.

异步适用于 IO 密集但计算量小的场景, 是一种绕过 IO 阻塞从而提高 CPU 利用率的解决方案. 常见的异步场景有资源加载和网络通讯.

异步的特点

异步的最大特点就是不会立刻完成, 而且是永远不会立刻完成. 任务量再小, 甚至是没有任务也不会立刻完成.

异步函数要不没有返回值并需要传入回调函数, 要不返回的是此刻做事的状态. 事情的状态一般也就是三种: 未开始, 正在做, 已完成. 当返回值显示任务已完成时就可以取得成果.

由于异步不能阻塞, 所以只能下发任务让其他人去执行, 同时需要派个人去盯着更新进度.

因为任务不会平白无故的下放, 任务要是完成了就会有后续需要做的事情, 所以需要回调函数.

既然如此, 异步编程的三大基本要素已经呼之欲出: 任务、执行者、监工.

GameFramework 中的异步编程

异步要素

  • 核心: TaskBase / ITaskAgent / TaskPool / ITaskAgentHelper
  • 辅助: StartTaskStatus / TaskStatus / TaskInfo

这里讲解一下辅助元素:

  • StartTaskStatus: Agent 开始执行任务时调用 Start 的返回值. 在任务开始前就对任务进行甄别, 避免无用消耗.
  • TaskStatus: 任务当前状态. 仅针对被执行的任务而言的三种状态: 未开始, 正在做, 已完成.
  • TaskInfo: 正如字面意思, 任务信息, 就是把任务的成员变量用结构体写了一遍.

核心元素之间的关系如下图:

  • ITaskAgent 是任务负责人
  • TaseBase 为任务本身, 提供必要的信息与存储 ITaskAgent 的成果信息
  • TaskPool 就是黑心工厂, 里面有个 Update 回调, 不断催促 ITaskAgent 完成手上的活并把剩下的活都给到 ITaskAgent
  • ITaskAgent 与 TaseBase 分别有子类, 子类是一一对应的关系; TaskPool不再有子类, 仅有一个泛型存在.
  • ITaskAgentHelper 是 ITaskAgent 借助的工具或帮手, 作为其成员变量而存在. 在 UGF 中实现此接口实现真正的功能.

里面还有一些内存管理方面的细节在这里就不展开了.

这里 TaskBase 和 ITaskAgent 是一一对应的关系, 每个 TaskBase 的子类都需要专门的 ITaskAgent 来处理. TaskBase 的状态由其所属的 ITaskAgent 决定.

E大声明了三种 ITaskAgent的子类, 分别处理对应的三种 TaskBase的子类:

  • DownloadManager.DownloadAgent <==> DownloadManager.DownloadTask

  • WebRequestManager.WebRequestAgent <==> WebRequestManager.WebRequestTask

  • ResourceManager.ResourceLoader.LoadResourceAgent <==> ResourceManager.ResourceLoader.LoadResourceTaskBase

至此, Task 体系就搭建好了. 之后要使用就调用 AddTask 即可. 是不是感觉很简单呢?

但——你是不是忘了什么重要的东西?

回调函数

为了让任务的成果不付之东流, 怎么能忘了它呢!!! 还是说你就是喜欢看别人忙活, 赚不赚钱不重要？？? 杀! 人! 诛! 心!

说回正题. 既然是任务完成的回调函数, 由执行任务的 ITaskAgent 来触发在合适不过了.

可是 ITaskAgent 没有任何回调相关的信息!!! 回调在哪儿呢? 在子类中.

为了减少干扰项, 以下仅选择”成功”的调用链进行说明

在 DownloadManager.DownloadAgent 中回调函数是 OnDownloadAgentHelperComplete

在 WebRequestManager.WebRequestAgent 中回调函数是 OnWebRequestAgentHelperComplete

这些回调函数在后文将被统称为 AgentOnFunction, 意思是声明在 Agent 中的以 On 开头的回调函数.

可是疑问还是没解决, ITaskAgent 不做实事, 这些回调函数什么时候被触发呢?

这里就轮到 ITaskAgentHelper 出场了.

通过查看 AgentOnFunction 的引用会发现这些代码

这些回调被加入了 ITaskAgentHelper 的事件之中, 而 ITaskAgentHelper 是接口, 我们也不必再去进一步跟踪了, 回调的触发者就是 ITaskAgentHelper 没错了.

在 GF 中并未使用 ITaskAgentHelper , 而 ITaskAgentHelper 的访问属性为 public 因此应该在 UGF 中才使用到.

果不其然, 在 UGF 中 DownloadAgentHelperBase 继承了 IDownloadAgentHelper; WebRequestAgentHelperBase 继承了 IWebRequestAgentHelper, 他们都使用了 Untiy 提供的 UnityWebRequest 来完成具体的任务. DownloadAgentHelperComplete 和 WebRequestAgentHelperComplete 就是在完成任务的时候被调用, 从而带动 OnDownloadAgentHelperComplete 和 OnWebRequestAgentHelperComplete.

目前我们能看到的调用链只有短短的 “IHelperEvent => AgentOnFunction”. 有头无尾.

继续跟进下去, 我们发现 AgentOnFunction 也有调用委托.

OnDownloadAgentHelperComplete 调用了 DownloadAgentSuccess .

OnWebRequestAgentHelperComplete 调用了 WebRequestAgentSuccess.

DownloadAgentSuccess 和 WebRequestAgentSuccess 分别是 DownloadManager.DownloadAgent 和 WebRequestManager.WebRequestAgent 中暴露给外部的委托. 以下统称为 AgentAction, 意思是声明在 Agent 中暴露给外部的委托.

而 AgentAction 又被谁加入了呢? 由于 DownloadManager.DownloadAgent 和 WebRequestManager.WebRequestAgent 均为类中类且访问权限均为 private 所以直接去对应的管理类看.

在 DownloadManager 中有一个函数 AddDownloadAgentHelper , 创建了 DownloadManager.DownloadAgent 并使自身的 OnDownloadAgentSuccess 监听 DownloadManager.DownloadAgent 的 DownloadAgentSuccess 委托.

而 OnDownloadAgentSuccess 又调用了 DownloadManager 的 m_DownloadSuccessEventHandler 委托. 虽然这个委托是 private, 但 DownloadManager 使用了 DownloadSuccess 事件将其公开出去了出去了.

这个 DownloadSuccess 声明在 IDownloadManager 中, DownloadManager只是实现了而已.

WebRequestManager 也是同理.

在 WebRequestManager 中有一个函数 AddWebRequestAgentHelper , 创建了 WebRequestManager.WebRequestAgent 并使自身的 OnWebRequestAgentSuccess 监听 WebRequestManager.WebRequestAgent 的 WebRequestAgentSuccess 委托.

而 OnWebRequestAgentSuccess 又调用了 WebRequestManager 的 m_WebRequestSuccessEventHandler 委托. 虽然这个委托是 private, 但 WebRequestManager 使用了 WebRequestSuccess 事件将其公开出去了出去了.

这个 WebRequestSuccess 声明在 IWebRequestManager 中, WebRequestManager只是实现了而已.

这里 OnDownloadAgentSuccess 和 OnWebRequestAgentSuccess 因为是定义在 Manager 中以 On 开头的函数, 以下统称 ManagerOnFunction. 同理 DownloadSuccess 和 WebRequestSuccess 因为是定义在 Manager 中的事件, 以下统称 ManagerEvent .

现在我们就获得了完成的调用链

IHelperEvent => AgentOnFunction => AgentAction => ManagerOnFunction => ManagerEvent => 外部回调委托.

由于 Manager 是公开且唯一的, 也将任务完成后的事件暴露出来了, 我们就能定义任务完成后该做什么了.

这次我们将回调函数也处理好了, 我们已经完整实现了异步的特点里阐述的所有要素, 但事情依然不简单. 其原因就在我上面故意漏掉的 Resource 环节里.

我们中出了一个叛徒

通过跟踪源码的调用逻辑, 这三种子类 ITaskAgent 的实际的调用链如下:

  • Download:
    IHelperEvent => AgentOnFunction => AgentAction => ManagerOnFunction => ManagerEvent => 外部回调委托
  • WebRequest:
    IHelperEvent => AgentOnFunction => AgentAction => ManagerOnFunction => ManagerEvent => 外部回调委托
  • Resource:
    IHelperEvent => AgentOnFunction => TaskOnFunction => TaskCallbacks => 外部回调委托

Resource 环节具体表现如下:

ILoadResourceAgentHelper 中的 LoadResourceAgentHelperLoadComplete 调用 ResourceManager.ResourceLoader.LoadResourceAgent 的 OnLoadResourceAgentHelperLoadComplete, 随后继续调用 OnAssetObjectReady, 再调用了 LoadResourceTaskBase 的 OnLoadAssetSuccess .

找不到的 TaskCallbacks 需要到 LoadResourceTaskBase 的子类中去找. 这个私有变量会在子类创建的时候传入, 里面是与 TaskOnFunction 一一对应的外部委托, 不同的 TaskOnFunction 会调用 TaskCallbacks 中的不同委托.

ILoadResourceAgentHelper 在 UGF 中被 LoadResourceAgentHelperBase 继承, 并在其子类中使用 Unity 提供的 AssetBundle 实现功能.

可 Resource 的调用链为何就不同呢. 通过调用链我们能清晰的认识到, Resource 是将外部回调委托存储在了 Task 上, 其他两类则是存储在了 Manager 里. 明明都是 Task 体系, 却为何产生了这种差异?

刚遇见这个问题的我简直炸裂, 在看到《StarForce》的项目演示里E大不遵守自己的规范后, 我第一时间想到的E大不遵守自己的规范, 第二是历史问题. 因为我没法解释怎么凭空产生了这么个差异.

不合理设计之一 : 回调归属问题

那么, 这个回调函数究竟是放在 Task 上还是 Manager 里呢?

先说结论, 我认为放在 Task 上才是正确的. 这里我得反省一下自己过分迷信E大了. 本来认为回调应该放在任务里的我, 因为跟踪的顺序是 Download => WebRequest => Resource, 合理化地认为将回调放在 Manager 是统一入口比较好. 而已经改变观念的我看见 Resource 的调用链有区别的第一个想到的居然是E大没遵守规范和历史问题, 完全忽略了在 Resource 链条里回调是存储在 Task 上的.

从常识角度上来说, 如果要说明某任务完成后的下一步行动, 放在任务本身的说明里肯定是最好的. 而且任务还没有就有了下一步行动并不合理.

注意 Task 和 Event 不同, 虽然二者都是基于委托. Event 与委托的对应早就存在, 只是等待 Event 发生. Task 不存在就有知道 Task 做完后该做什么属于违反因果律.

Event 可以类比成需要触发等待的任务, 而这里的 Task 本身就是为了解决异步的等待问题, 需要立即执行.

此处用英文表述的指代码层面的意思, 中文的任务代表日常逻辑中的意思.

从代码的角度上来看呢?

让我们试试如果将其他两个改成 Resource 模式会是怎样的.

首先让我们看一下 Download 和 WebRequest 对于 Task 的使用:

  • 原版的调用链是 IHelperEvent => AgentOnFunction => AgentAction => ManagerOnFunction => ManagerEvent => 外部回调委托

  • Task 的创建全在 Manager 之中, 也就是创建时能获取到 ManagerOnFunction

  • AgentAction 只有 Manager 在使用, 也就是 AgentAction 完全是为了调用 ManagerOnFunction

于是可知 TaskOnFunction 完全可以替代 AgentAction, 从而将调用链改成如下模式:

IHelperEvent => AgentOnFunction => TaskOnFunction => TaskCallbacks => ManagerOnFunction => ManagerEvent => 外部回调委托.

因为有了 TaskCallbacks, 后面的不再需要, 去掉那些多余的部分, 就变成了和 Resource 一样的调用链:

IHelperEvent => AgentOnFunction => TaskOnFunction => TaskCallbacks => 外部回调委托.

一个比原调用链更短且代码可读性更高的调用链.

但是会如果是把 Resource 调用链替换成另外两个的类型, 最终就需要在一个回调入口同时适配 EntityManager / SoundManager 和 UIManager. 至少唯一可求证的 Download 是这么做的.

在 GF 中唯二使用 DownloadManager 执行下载函数的 ResourceManager.ResourceUpdater 和 ResourceManager.VersionListProcessor 中均对类型转换做了成功与否的判定, 从而确定是不是自己应该响应的回调, 而 Resource 逻辑下完全没有这一步.

这个判断方法有一种缺陷: 无法辨别是不属于自己应该响应的回调还是传参不正确. 在 Resource 体系下就完全不会出现这种情况, 是 null 就一定是传参不正确. 不可能是自己不该响应的.

再就是这种由多个地方聚拢再分流的策略对性能会产生影响. 明明每一次触发回调只有其中一个能响应, 却回调了所有监听的委托, 即使在开头就进行了判定.

而且如果 Download 也是在任务建立时传入回调函数的话, ResourceManager 会变的更加简洁.

不合理设计之二 : AgentOnFunction 未解之谜

在跟踪回调中发现一个问题, 在 AgentOnFunction 中既然 GameFrameworkEventArgs 已经停止传递为何没回收? 这里是引用池内存泄漏了吗?

这里以 OnDownloadAgentHelperComplete 函数举例说明

不过最终我在 UGF 处找到了答案.

E大将创建和回收放在了一起. 创建与回收放一起就一目了然.

然后更诡异的事情出现了.

这是 HelperEventAction:

event EventHandler<DownloadAgentHelperCompleteEventArgs> DownloadAgentHelperComplete;

这是 AgentAction:

GameFrameworkAction<DownloadAgent, long> DownloadAgentSuccess;

这是 ManagerEvent :

event EventHandler<DownloadSuccessEventArgs> DownloadSuccess;

这是调用链:

IHelperEvent => AgentOnFunction => AgentAction => ManagerOnFunction => ManagerEvent => 外部回调委托

为何 IHelperEvent 和 ManagerEvent 都使用了 EventHandler<GameFrameworkEventArgs> , 偏偏处于二者之间的 AgentAction 改用了 GameFrameworkAction. 我至今也没想明白. 明明可以一直传下去却不传, 离谱.

TaskBase

TaskBase 就是在记录任务有关的一切信息, 包括任务下达时的信息和任务进行中的信息.

只简单提三点:

  • Task 有优先级设定, 与 GameFrameworkModule 是同等的设定.
  • m_SerialId 成员是任务的唯一标识, 仅在同类任务中唯一. GF 中是通过子类的静态变量自增保证唯一的.
  • m_UserData 会在任务结束后使用到, 具体使用与否看你自己. (不知怎么地我就不敢用 object 类型, 我好菜, 呜呜呜呜)

但有个地方我就不太理解, 整个 GF 里关于 TaskInfo 的使用全都是 new TaskInfo(task.xxxx,task.xxxx,…) 甚至还用 task.Done 来判断 TaskStatus 应该显示成什么. 所以, TaskStatus 为什么没有成为 TaskBase 的成员变量呢, 我不理解, 反正我是放进去了, 然后把 Task.Done 删了. 还加了个 bool CompareTag(string tag) 函数优化性能, 再加个 TaskInfo GetTaskInfo() 减少工作量, 现在感觉很舒适.

任务与事件的对比

比对项 Task Event
执行时机 因任务耗时延迟执行 等待事件延迟执行
回调生命周期 与 Task 保持一致 与订阅者保持一致
回调访问权限 私密(仅任务发布者可知) 公开(可由任何对象订阅)

知识补充: 代理模式——感谢檻鹤

为其他对象提供一种代理以控制对这个对象的访问。在某些情况下，一个对象不适合或者不能直接引用另一个对象，而代理对象可以在客户端和目标对象之间起到中介的作用。

  • 抽象角色：通过接口或抽象类声明真实角色实现的业务方法。

  • 代理角色：实现抽象角色，是真实角色的代理，通过真实角色的业务逻辑方法来实现抽象方法，并可以附加自己的操作。

  • 真实角色：实现抽象角色，定义真实角色所要实现的业务逻辑，供代理角色调用。

这三种角色在 GF 里分别是:

  • 抽象角色 —— IHelper
  • 代理角色 —— ITaskAgent
  • 真实角色 —— UGF 中的 Helper 类

但是E大仅使用了代理模式的思想并未使用代理模式的形式.

Helper 类继承了 IHelper 但 ITaskAgent 并未继承 IHelper. 不符合代理模式的形式, 但依然是通过抽象角色间接调用了真实角色. E大用子类变体的方式使得在保持公开方法不变的情况下巧妙地通过 IHelper 调用真实角色完成了任务.

代理模式的优点是能将代理对象与真实对象被调用的目标对象分离, 能保护真实对象, 并在不破坏真实对象的基础上进行方法拓展.

这种隔离性好的优点被反向利用就有了只定义接口等待外部继承实现的用法, 正因为隔离性好才让实现与使用分离成为了可能.

不过讲道理, 代理类真没必要和真实类的方法强行保持一致. 这种思想非常好, 但形式有点过于鸡肋了.

意外收获

在本次的探索过程中发现了一个额外的惊喜. 就是部分在 GF 中定义的 EventArgs 在 UGF 中又定义了一遍, 只不过基类从 GameFrameworkEventArgs 变成了 GameEventArgs . 因为E大要将这些事件用事件中心传出去.

我在写事件中心篇的时候就说过相关情况, 当时看到基类是 GameFrameworkEventArgs 就认为是E大为了防止能用事件中心传出去, 我以为是E大不准备传出去, 最后依然传出去了, 只是在我还没关注到的 UGF 里(笑). 在我看来完全可以把框架事件直接作为游戏事件, 因为框架事件在 UGF 层被转换成了游戏事件. 而走事件中心无疑是让回调路径最短且最自由的做法. 从内存上来说是节约了一个静态整型变量, 从访问权限上来说并没有得到控制, 从使用上来说变得更加复杂了. 在我看来为了区分这两种类型让回调路径变长且跟踪困难还把类变得复杂并且声明了多余的类和实例化了更多的类是得不偿失的.
