Game Framework - Web请求与下载

发表于2023-01-30|更新于2023-04-02|C#Game Framework
|字数总计:4.8k|阅读时长:15分钟|阅读量:

文章范围

本篇文章会包含 GF 的 Download 和 WebRequest 两个文件夹的内容. 至于为什么会放在一起讲, 原因有三.

  1. 这两个在 UGF 中使用的是同一个工具 UnityWebRequest 实现

  2. 两者同属于异步

  3. 两者的设计出的问题也是一模一样

UnityWebRequest

由于我不太懂 UnityWebRequest, 所以先学一会儿, 已经会了的可以跳过这一节.

学习参考: UnityWebRequest - Unity 手册 (unity3d.com)

UnityWebRequest 分为高级 API 和低级 API, 此处的高级和低级代表封装程度. 高级 API 都能使用低级 API 实现. 对于简单需求只需要使用高级 API, 因为高级 API 仅是省略重复代码而存在. 低级 API 更加通用.

常用高级 API 有 Get 和 Post. Get 相当于实例化后赋予了 DownloadHandlerBuffer 后发送请求; Post 相当于实例化后赋予了 UpdateHandlerRow 后发送请求;

而涉及到 Header 设置的部分则只能通过低级 API 实现. 因为仅在实例化后才有 SetRequestHeader 函数.

Header 的设置在网页端再常见不过了, 例如最基本的 UserAgent 字段.

在之前我不懂为什么 Download 要实例化而 WebRequest 不实例化的原因. 现在我知道了. 因为 Dowload 使用的时候使用了 SetRequestHeader, 而 WebRequest 没有.

Unity 为各种资源都专门配置了获取方法, 例如纹理 / AB包 / 音频 / 视频, Unity 将这些方法经过了专门的优化, 比自己手动获取数据后转换的表现更加出色.

在此知识点 Unity 官方文档有问题.

截止 2023年1月23日21:40:06, Unity 2020.3 官方文档都写的是

  • “要从远程服务器下载 AssetBundle，可使用 UnityWebRequest.GetAssetBundle。此功能将数据串流到内部缓冲区，后者负责在工作线程上解码和解压缩 AssetBundle 的数据。”
  • “要从远程服务器检索纹理文件，可使用 UnityWebRequest.Texture。此函数与 UnityWebRequest.GET 非常类似，但进行了优化，可高效下载和存储纹理。”

但实际在 Unity 2020.3 版本上的使用方法是:

  • UnityWebRequestAssetBundle.GetAssetBundle
  • UnityWebRequestTexture.GetTexture
  • UnityWebRequestMultimedia.GetAudioClip
  • UnityWebRequestMultimedia.GetMovieTexture

下面两种在官方文档中并未提及, 但在 创建 DownloadHandler - Unity 手册 (unity3d.com) 页面能看到对应的 DownloadHandler, 我认为必定有其对应的 API, 所以我找出来了.

虽然涉及到两个文件夹, 但在有了上一篇文章的基础上却一点都不复杂, 所有的设计都是非常直接的按需设计.

Download

DownloadTask

DownloadTask 继承自 TaskBase, 依赖于 DownloadTaskStatus. 这里占据的 TaskStatus 想必就是E大不将 TaskStatus 作为成员变量的原因. 但这个理由是站不住脚的, 因为整个 GF 声明的作为专有成员变量的 TaskStatus 仅有 DownloadTaskStatus 和 WebRequestTaskStatus, 这两个成员变量还是如上篇所说完全可以被加入 Error 状态的 TaskStatus 替代, 更何况将 Error 状态加入 TaskStatus 本身就合情合理.

DownloadTask 记录了 m_DownloadUri(下载源), m_DownloadPath(文件存放路径), m_FlushSize(缓冲区长度) 和 m_Timeout(超时时间).

值得一提的是 s_Serial 为 DownloadTask 的静态私有变量, 每创建一个 DownloadTask 便会自增并存储到 TaskBase 的私有成员变量 m_SerialId, 从而区分每一个 DownloadTask , 这个 ID 在 DownloadAgent 的范围内是唯一的.

DownloadAgent

这个类实现 IDisposable 接口的原因只有一个, 其成员变量有一个 FileStream 类型的成员变量. 接口的作用就是在这个类释放的时候一并释放 m_FileStream.

m_WaitFlushSize 和 m_WaitTime 与 m_DownloadTask 中成员变量的关系就是配置数据与运行时数据的关系.

m_StartLength / m_DownloadedLength / m_SavedLength 是相互关联的.

  • m_StartLength 用于断点续传, 在任务开始时读取文件字节长度, 若文件不存在则为0
  • m_DownloadedLength 用于记录下载字节长度
  • m_SavedLength 可以改名 m_FlushedLength , 指已经写入硬盘的长度. 在文件下载完毕的时候会与 m_StartLength + m_DownloadedLength 的结果比较判断是否下载错误.

存储一定的字节数再写入硬盘有利于提高性能, 因为与硬盘交互的速度要比内存慢很多.

还需要注意 m_WaitTime 是有在 OnDownloadAgentHelperUpdateBytes / OnDownloadAgentHelperUpdateLength / OnDownloadAgentHelperComplete 重置的

这个类里有个显而易见的问题, private void Dispose(bool disposing), 这个函数很多余.

关于 GC.SuppressFinalize 我也不太懂, 参考这篇文章 从GC的SuppressFinalize方法带你深刻认识Finalize底层运行机制 - 一线码农 - 博客园 (cnblogs.com)

IDownloadAgentHelper

IDownloadAgentHelper 的下载功能由 UnityWebRequest 提供, 数据更新事件由 DownloadHandler 提供, 下载完成与下载错误由 UnityWebRequestDownloadAgentHelper 在 Update 函数中提供.

至于 UnityWebRequestDownloadAgentHelper 的 Update 函数为啥没问题, 是因为在 DownloadAgent 的 OnDownloadAgentHelperComplete 里调用了其 Reset 函数使 m_UnityWebRequest 变量为 null.

实现下载器的通用功能

在介绍下载器之前, 我们先观察一下市面上已有的下载器特性.

  1. 可以同时下载多个文件
  2. 可以显示下载文件信息, 包括名称和文件大小
  3. 可以显示文件下载速度和下载进度
  4. 可以断点续传

这些特性有些 GF 已经提供而有些没有. 接下来我会将 UGF 和 GF 联合起来进行分析, 看看 GF 是如何实现这些功能的.

现成的信息

UnityWebRequest 为我们提供了:

  • downloadedBytes / uploadedBytes 用于知道已经传输的字节长度
  • downloadedProgress / uploadedProgress 用于知道传输进度
  • isDone 用于判断是否完成

显然, 我们能从 UnityWebRequest 中得到的只有文件下载速度和下载进度, 其他的都需要我们手动实现.

实现断点续传

断点续传在了解之前觉得是一个好高级好高级的玩意, 但看到其实现代码后才发现原来如此简单.

断点续传的实现需要有两个前提:

  1. 下载器支持断点续传: 指下载器需要能告知服务器从哪里开始下载.
  2. 服务器支持断点续传: 指能处理客户端发来的断点续传的信息.
  3. 关于这两个前提的有力证据就是 IDM 下载器. IDM 本身支持断点续传, 并且在下载窗口会显示本次下载是否支持断点续传. 笔者遇到过几次是否支持断点续传处写否的情况, 由此可知断点续传需要客户端与服务端都提供对应的功能才能实现.

基于以上这两点, IDownloadAgentHelper 中 Download 函数具有 fromPosition 和 toPosition 这两个参数就不再显得奇怪了. 将画面切到 UGF 中看其具体的实现, 是在请求头中设置了 Range 头文件信息告知服务器本次需要下载的数据范围. 服务器只需要将数据范围内的字节传递给客户端, 客户端接收到数据后添加在文件末尾即可实现断点续传了.

PS: 断点续传配合异步多线程能实现一个文件分段下载. 参考 IDM 的做法.

实现计算下载速度

减法获取

UnityWebRequest 提供的 downloadedBytes 可以通过当前值减去上一帧的值获得当前帧下载的字节长度从而获得当前的下载速度.

重写 DownloadHandlerScript 获取

在 UGF 中重写 DownloadHandlerScript 类中的 ReceiveData 函数. 此函数每帧触发, 将当前的字节数组和数组长度作为参数传入此函数. 免去了减的步骤, 也相比于减法步骤更加可靠. UGF 额外在此函数中触发了 DownloadAgentHelperUpdateBytesEventArgs 和 DownloadAgentHelperUpdateLengthEventArgs 事件, 从而开始了事件传递.

实现显示文件信息

GetRequestHeader 函数

文件信息只能从服务器获取, 并不属于客户端的功能, 因此这里就是和服务端商量好协议才能实现此功能.

在 Http 协议中资源大小一般以 Content-Length 字段传递, 因此可以通过 GetResponseHeader("Content-Length") 获取到资源的大小. 文件名的其他信息同理, 以头文件传递让客户提前知晓即可.

重写 ReceiveContentLength 获取

在 DownloadHandlerScript 类中也有对应函数 ReceiveContentLength ,此函数会在收到 Content-Length 标头时调用, 需要特别注意的是: 如果服务器在处理 UnityWebRequest 的过程中发送一个或多个重定向响应，则此回调可能会多次发生。

实现多文件下载

单 MonoBehavioiur 协程

对于有异步支持的 Unity 来说，多文件同时下载并不算难事, 只需要协程等待即可. 此方法只需要一个 GameObject 挂载一个 MonoBehaviour.

多 MonoBehavioiur 轮询

UGF 则是采取的此方式. 需要几个文件同时下载就实例化几个 DownloadAgentHelper, 然后在 Update 生命周期函数中轮询是否已完成下载.

不过关于是否已完成下载也可以通过重写 DownloadHandlerScript 类中的 OnContentComplete 实现. 当 UnityWebRequest 完全从服务器下载所有数据并且已将所有接收的数据转发到 ReceiveData 回调时会自动调用此函数.

性能优化

预提供字节数组

DownloadHanlder 的 ReceiveData 函数会在每次调用时实例化一个数组, 因此下载需求量大的时候会产生大量GC, Unity 提供了预提供字节数组的办法来解决这个问题, 缺点就是每次传的数据量不能大于这个字节数组的容量, 不过这个缺点其实并不算什么问题. 使用办法就是在实例化 DownloadHandler 的时候把这个预先生成的字节数组传进去.

通过枚举参数使用Unity优化过的下载函数

上面提到了 Unity 专门为各种特定类型的文件提供了优化过后的函数, 效率和性能比自己转换高出不少. 当然不嫌麻烦的话也可以根据官网上的说明自己把这个优化给实现了.

DownloadCounter

这个类不是下载计数器, 是下载计速器. 不是每一个任务的, 而是所有任务总和的.

虽然之前能得到下载速度, 但是其更新频率过高且下载速度是一个不稳定的数值, 所以不适合人去直观了解下载速度, 甚至人可能看到的都是数字的残影. 所以需要专门做一个能间隔固定时间更新下载速度的计算类.

我们都知道, 下载速度就是在单位时间里的字节下载量. 实现也无非就是, 每隔多少秒执行一次计算. 但是打开代码一看, 我看不懂了. 这么简单的问题为什么会写的这么复杂.

首先看一下其私有类 DownloadCounterNode , 这是一个只有两个成员变量的数据类. m_DeltaLength 记录下载的字节长度, m_ElapseSeconds 记录节点从节点的存在时间. 然后其 Update 函数仅仅是在增加存在时间, AddDeltaLenth 函数仅仅是增加字节下载量. 是一个非常简单的类.

再来看看其本体, 这个里面值得关注的就两个函数 RecordDeltaLength 和 Update. 需要注意的是任意一个 DownloadAgent 发生数据传输都会触发 RecordDeltaLength , Update 仅在 DownloadManager 的 Update 中调用, 因此两个函数的调用频率并不同步.

RecordDeltaLength 函数

找到其被调用的地方, 发现仅在 OnDownloadAgentUpdate 里被调用, 而 OnDownloadAgentUpdate 函数的 deltaLength参数是传递的下载的字节长度. 由此可知此函数是记录当前网络传输包的字节量.

但是上面的代码的可读性在我看来非常差, 以至于这注释我都修改了好几次, 于是我将之重写成了这样.

Update 函数

这个函数就更抽象了, 看的我人都是晕的, 不能理解. 每一段我都懂, 但连起来我就看不懂了, 但是理清思路之后代码就变得清晰了.

  1. 所谓实时的下载速度就是最近的下载量除以时间.

  2. 最近的下载量是如何体现的呢? 当节点越过规定的时间后回收使其失效. 从而不计入统计环节.

  3. 时间是什么呢? m_Accumulator 变量所记录的值. 这个值没有任何重置的地方, 那么也就是其最大值为 m_RecordInterval. 一般就等于 m_RecordInterval

基于上述理解, 我将代码修改成了这个样子, 可理解的程度得到相当大的提升.

要特别注意的是处理节点生命周期的那两个循环并不能合并. 因为 foreach 循环不能在循环中修改迭代器, while 循环中会 break 导致不能更新所有节点的存在时间.

DownloadEventArgs

这里额外说一下 DownloadStartEventArgs / DownloadSuccessEventArgs / DownloadUpdateEventArgs / DownloadFailureEventArgs 这四个是留个外部使用的, 在 DownloadManager 中的成员事件使用.

再就是这几个事件在 UGF 中又定义了一遍, 用于事件中心发布事件.

WebRequest

E大是使用的是高级 API 实现, 但因为高级 API 可以通过低级 API 实现, 所以基本和 Download 相同, 甚至更加简单, 只不过是请求请求网页而已.

自己在 UGF 中將 WebRequest 通过低级 API 实现出来也可以.

如果需要专门写出来的话请向我反馈, 我后续再进行补充更新.

此次设计出了什么问题

  • 设计了冗余的任务状态类

    E大为 Download 专门写了一个 DownloadTaskStatus ,又为 WebRequest 专门写了一个 WebRequestTaskStatus 其本质只是比 TaskStatus 多了一个 Error 状态的枚举, 而此状态也完全可以加入 TaskStatus. 并且这两个任务的标志位也完全可以通过 TaskStatus 来代替.

  • DownloadHelper 没有信息要在 Inspector 上显示却单独挂载了脚本并使用其 Update 函数

    在生命周期篇我们就讲过在减少 GameObject 的使用和白嫖生命周期函数对于提高性能的重要意义, 此处在 Inspector 窗口没有任何显示却使用了 Update 这个频繁调用的生命周期的函数让我不太能理解. 不过当我从合理性寻找的时候发现了一些我思想上的盲区, E大设计的接口有一个原则: 不希望使用者知道的函数永远不出现在接口中, 这样很安全保险. 这个思想也解决了我之前的一个疑惑, 又希望能被外部调用但不希望被别乱调用的设计方法.

  • IDonwloadHelper 的 Download 函数声明了从不使用的参数 userdata

    在 UGF 中可以看到这个 userdata 完全没有被使用. 这个地方想要变得合理就需要对 UGF 本身进行改动, 将 userData 转换成自定义的类然后将其中的值给 m_UnityWebRequest 使用, 例如 m_UnityWebRequest.SetRequestHeader.

  • 完全没考虑网络攻防的问题

    考虑到 GF 将重心放在客户端所以也能理解. 这也不是一个简单能解决的问题, 网络这一块去关注 ET 框架比较好, E大将实现简化至此反而更方便我们学习.

  • DownloadAgentHelperUpdateLengthEventArgs 事件没必要单独出来

    事实上此事件的作用完全被 DownloadAgentHelperUpdateBytesEventArgs 包含了. 也因此可以删掉相关的一些事件委托和一些回调函数.

  • DownloadAgent 没必要继承 IDisposable 接口

    事实上除了自带的 Shutdown 函数没有地方调用此接口, DownloadAgent 也并未设计到 using 用法. 所以其 Dispose 函数仅存的意义就是释放掉其成员变量 m_FileStream , 而这个完全可以直接写在 Shutdown 函数中.
