Game Framework - 内存管理

发表于2022-11-21|更新于2023-03-15|UnityGame Framework
|字数总计:3.1k|阅读时长:10分钟|阅读量:

在上一篇中, 我们解决了生命周期的难题, 使 C# 类也能白嫖到 Unity 引擎的生命周期, 节约了内存也节约了性能. 更是使用了管理模式来统一入口和管理内存, 本篇中我们就来细说内存管理.

C# 中的 GC(Garbage Collection)

GC, 即垃圾回收器, 用以辅助程序对用不到的内存进行回收再利用. 在 C++ 中, 内存的申请与释放都需要手动进行, 很容易造成资源浪费与内存泄漏. 为解决这个问题, 目前主流语言都采用了 GC 这种内存管理策略.

GC 的管理策略即在程序初始化的时候就申请一大块内存作为内存池, 程序中的几乎所有变量都会存在这个内存池中, 每当需要申请内存时也是从内存池中获取. 使得程序做到对内存的几乎完全控制. 为什么是几乎? 因为有 unsafe 写法的存在.

C# 中的 GC 在第一代 GC 装满时触发, 在对第一代 GC 中所有对象遍历后回收未被使用的内存, 并将仍在使用的内存移植到第二代 GC, 如果第二代 GC 满了也会再次迁移到第三代 GC. 既然 GC 产生时会对大块内存进行遍历, 自然会是一个耗时又耗性能的工作. 一般来说, 游戏开发者希望 GC 只发生在过场景读条的阶段, 这样对玩家的影响最小, 除此之外的任何 GC 都不是开发者希望看到的.

如何阻止 GC 的产生?

在说 GC 前首先得知道 C# 的两种变量类型: 值类型和引用类型. 值类型放在栈内存中, 在变量生命周期结束时由系统自动释放. 引用类型放在堆内存中, 由 GC 管理. 既然知道 GC 管理的是引用类型, 那么创建引用类型就会给 GC 增添垃圾, 第一代 GC 被填满就会触发 GC, 所以减少引用类型的创建就成了控制 GC 的不二之选, ReferencePool 应运而生.

ReferencePool(引用池)

引用池可以理解为自己在运行时覆盖的一层垃圾回收机制, 由程序员自己通过操作引用池重复利用已生成的引用类型对象. 引用池有五个必备要素: 入池(模拟释放内存), 出池(模拟申请内存), 新增, 移除, 回收接口(在回收时执行的函数).

GF 既然将 ReferencePool 作为静态管理类型, 经过上一篇的折磨我们已经能直接说出答案了 —— ReferencePool 是管理”管理类”的管理类. 其中 ReferenceCollection 就是管理类, 每个 ReferenceCollection 都是一种类型的管理类, ReferencePool 通过 Type 索引到 真正的 ReferenceCollection, 委托 ReferenceCollection 进行实际操作.

ReferenceCollection(引用集合)

引用集合是引用池的实际执行者，所有引用集合被引用池管理， 负责某一类(class)的内存管理.

ReferencePool的使用

直接使用

优点:

  1. 使用方便快捷, 即写即用.

缺点:

  1. 写的代码会比较多
  2. 无法很好的应对需要初始化的类型(这个将在下面讲解)

函数使用

一个类创建出来往往需要参数, 这原本是构造函数的职责. 但是因为引用池的限制, 不得不废掉构造函数的这个作用. 这种条件多了之后就容易记不住造成初始化未完成程序报错. 于是在 Girl 里声明一个静态函数用于创建 Girl, 函数参数就写必须初始化的成员变量, 这就是 GF 所使用的方法.

优点:

  1. 使用时代码少
  2. 能完美解决初始化问题

缺点:

  1. 使用前必须要声明一个静态函数

GenericVariable : 突破引用池的使用限制

使用 ReferencePool 的类型有三个要求:

  1. 是引用类型
  2. 继承 IReference 接口
  3. 具有无参公共构造函数

对于不满足这三类的就无法通过 ReferencePool 进行管理. 需要另寻他路.(笔者一直在想能不能兼容没有无参公共构造函数的, 因为希望只有管理器才能创建, 死了一堆脑细胞, 这还仅仅是为了满足一点)

可选的内存管理策略

此节的内存管理策略可用在任何管理类, 并不局限于 ReferencePool

  • 回收模式：

    指如何在需要的时候从对象池中取出对象，在不需要的时候将对象放回对象池

    • 借用：将对象从对象池中借用出来，借用后在对象池中找不到对象的引用。借用者不需要的时候将对象返回给对象池，借用者不再持有对象的引用。这种方式实现起来简单，大多数对象池都是用这种方实现，本文也采用这种方式实现。
    • 引用计数：用于同时有多个借用者访问同一个对象，只有当所有的借用者都释放了对象引用时，对象才可以被回收。每个对象都持有一个内部计数器和一个指向池的引用。当计数器为 0 时，对象就会返回池中。也即对象需要额外挂载一个脚本。

  • 分配方式：

    指对象池内的没有可用对象时，有新的请求时，该如何返回一个对象，常见的策略有：

    • 拒绝请求：告知池内没有对象了，拒绝请求，返回空对象
    • 强制回收：强行回收一个已经借出的对象
    • 增大池子：这种方式使用最多，什么时候增大池子呢？有三种触发方式
      • 空池触发：对象池内没有可用对象时，创建一个新对象
      • 水位线触发：空池触发的缺点是，请求对象时可能会因为执行对象分配而中断。为了避免这种情况，可以使用水位线触发。当从池中可用对象小于某个阈值，例如池子大小的十分之一，就触发分配过程。
      • Lease/Return速度：大多数时候，水位线触发已经足够，但有时候可能会需要更高的精度。在这种情况下，可以使用lease和return速度。例如，如果池中有100个对象，每秒有20个对象被取走，但只有10个对象返回，那么9秒后池就空了。开发者可以使用这种信息，提前做好对象分配计划。

  • 增长策略：

    指如何增大池子

    • 固定增长：每次增长一个或几个
    • 快速增长：根据池子大小的百分比增长

知识点图

至此, ReferencePool 貌似已经解决了内存问题, 但实际上还远没有结束. Reference 的管理策略适合需要反复创建使用的, 对于不需要反复创建使用的则不太适合. 强行使用会浪费一些内存和性能. 对于只需要创建一次的东西, 可以使用单例模式, 也可以如我们第一篇所说, 通过 “管理管理类的管理类” 进行管理, 保证其全局唯一.
